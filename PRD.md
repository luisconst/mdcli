# PRD: Browser Session Authentication for mdcli

## Overview

Add a new default authentication method that extracts the MeuDinheiro session from the user's existing browser profile. This eliminates the need for 1Password setup or manual login for users already authenticated in their browser.

### Problem Statement

Current authentication methods require either:
- **1Password integration**: Requires 1Password CLI + item setup
- **Browser login**: Opens new browser, requires manual login
- **Manual entry**: Copy-paste 5 headers from dev tools

Users who are already logged into MeuDinheiro in their browser have valid sessions that the CLI cannot access.

### Solution

Launch a browser with the user's existing Chrome/Firefox profile, navigate to MeuDinheiro, and extract authentication credentials from `window.loginconfig` (server-injected configuration).

---

## Goals

1. **Zero-friction auth** for users already logged into MeuDinheiro in their browser
2. **Make browser session the new default** auth method
3. **Support Chrome and Firefox** profiles
4. **Maintain backward compatibility** with existing auth methods

## Non-Goals

- Reverse-engineering Angular's header generation logic (NOT POSSIBLE - see Technical Discovery)
- Supporting Safari or other browsers (out of scope)
- Storing browser profile paths in config (auto-detect only)
- Supporting remote browsers or Docker environments

## Technical Discovery

### Key Finding: Headers are SERVER-GENERATED

Investigation revealed that `mdapikey`, `mdpolicy`, `mdsignature` are **generated by the server** and injected into the HTML via `window.loginconfig`. The Angular client does NOT compute these values—it reads them from the server-rendered page.

**Header Structure:**

| Header | Format | Example | Regenerated on Page Load |
|--------|--------|---------|--------------------------|
| `mdApiKey` | Hex string (20 chars) | `775b1940a83eb7663c43` | No (static per session) |
| `mdPolicy` | Base64(JSON) | `eyJrZXkiOi...` → `{"key":"775b...","expiration":"20260124T195913Z"}` | Yes (new expiration) |
| `mdSignature` | Base64(HMAC-SHA1) | `G6GOY8ExF62Du/Kone7m8S33Tts=` (28 chars) | Yes (recomputed) |
| `mduid` | Integer string | `430631` | No (user ID) |

**Signature Algorithm (Server-Side):**
```
mdSignature = Base64(HMAC-SHA1(mdPolicy, SECRET_KEY))
```
- Uses AWS S3-style authentication (Signature Version 2)
- SECRET_KEY is only on the server (cannot be reverse-engineered)
- 28-character Base64 output = 20-byte HMAC-SHA1 digest

**Implication:** We CANNOT generate signatures ourselves. We MUST extract them from the server-rendered page.

## Technical Approach

### Why Browser Launch is Required

| Approach | Viability | Reason |
|----------|-----------|--------|
| Read cookies from disk | **Not viable** | HttpOnly cookies cannot be read |
| Reverse-engineer signature | **Not viable** | Server secret is hidden |
| HTTP fetch without browser | **Not viable** | Need session cookies for authenticated page |
| Launch browser + extract config | **Viable** | Server renders `window.loginconfig` with valid credentials |

### Optimized Extraction (No Request Interception Needed)

Instead of waiting for API requests, we extract `window.loginconfig` directly from the page:

```
┌─────────────────────────────────────────────────────────────────┐
│  mdcli auth login [--session chrome|firefox]                    │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  browser-session.ts                                             │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │ 1. Copy Chrome/Firefox profile to temp directory           │ │
│  │ 2. Launch Playwright with copied profile                   │ │
│  │ 3. Navigate to MeuDinheiro                                 │ │
│  │ 4. Extract window.loginconfig via page.evaluate()          │ │
│  │    - mdApiKey, mdPolicy, mdSignature available immediately │ │
│  │ 5. Extract JWT token from cookies (for uid + token)        │ │
│  │ 6. Close browser + cleanup temp directory                  │ │
│  └────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  AuthConfig saved to ~/.config/mdcli/mdcli.config.json          │
└─────────────────────────────────────────────────────────────────┘
```

**Advantages over Request Interception:**
- Faster: No need to wait for API requests
- Simpler: Direct JavaScript evaluation
- More reliable: Config is always present on page load

### Profile Paths

| Platform | Chrome | Firefox |
|----------|--------|---------|
| macOS | `~/Library/Application Support/Google/Chrome` | `~/Library/Application Support/Firefox/Profiles` |
| Windows | `%LOCALAPPDATA%\Google\Chrome\User Data` | `%APPDATA%\Mozilla\Firefox\Profiles` |
| Linux | `~/.config/google-chrome` | `~/.mozilla/firefox` |

## Appendix: Header Structure Reference

### window.loginconfig Object

The server injects authentication data into the page as `window.loginconfig`:

```javascript
window.loginconfig = {
  "mdApiKey": "775b1940a83eb7663c43",           // 20-char hex string (static)
  "mdPolicy": "eyJrZXkiOi...%3D",               // URL-encoded Base64 JSON
  "mdSignature": "G6GOY8ExF62Du%2FKone7m8S33Tts%3D", // URL-encoded HMAC-SHA1
  "uid": 430631,                                // User ID (null if not logged in)
  "email": "user@example.com",
  "criando": false,
  "vinculando": false,
  "moedas": [...]                               // Currency list
}
```

### Decoded Policy

```json
{
  "key": "775b1940a83eb7663c43",
  "expiration": "20260124T195913Z"   // ISO 8601 UTC timestamp
}
```

### Signature Algorithm

- **Type**: HMAC-SHA1 (AWS S3 Signature Version 2 style)
- **Input**: The `mdPolicy` value (Base64 JSON)
- **Secret**: Server-side only (cannot be extracted)
- **Output**: 20-byte hash → 28-char Base64 string

### Mapping to AuthConfig

| loginconfig | AuthConfig | Notes |
|-------------|------------|-------|
| `mdApiKey` | `apiKey` | Direct mapping |
| `mdPolicy` | `policy` | URL-decode before storing |
| `mdSignature` | `signature` | URL-decode before storing |
| `uid` | `uid` | Convert to string |
| (cookie `mdauthtoken0`) | `token` | JWT from HttpOnly cookie |

### Discovery: `loginconfig` Behavior Across Pages

Browser testing revealed that `window.loginconfig` behaves differently than initially documented:

| Page | URL | `loginconfig` Present | Structure |
|------|-----|----------------------|-----------|
| **Login page** | `https://app.meudinheiroweb.com.br/` | ✅ Yes | `uid: null`, `mdApiKey`, `mdPolicy`, `mdSignature`, `moedas` |
| **MFA page** | `https://app.meudinheiroweb.com.br/u/{uid}` | ✅ Yes | `mdauthtoken` (JWT), `email`, `mdApiKey`, `mdPolicy`, `mdSignature` — **NO `uid` field** |
| **Dashboard** | `https://app.meudinheiroweb.com.br/u/{uid}/visaogeral` | ❌ No | `undefined` |

**Key Finding**: `loginconfig` is **NOT set by an API request**. It is server-side rendered directly into the HTML. The relevant API endpoints (`/api/v1/usuarios/login`, `/api/v1/usuarios/mfa`) return page redirects, not JSON.

### MFA Page `loginconfig` Structure

```javascript
window.loginconfig = {
  "mdApiKey": "775b1940a83eb7663c43",
  "mdPolicy": "eyJrZXkiOi...%3D",
  "mdSignature": "8kFswy94oEyjd8NSuDM%2Bn8DpVPI%3D",
  "mdauthtoken": "eyJ0eXAiOiJKV1QiLC...",  // JWT token directly available
  "email": "user@example.com",
  "isMfaEmail": false
  // NOTE: No `uid` field - extract from JWT's `uids` claim or URL
}
```
